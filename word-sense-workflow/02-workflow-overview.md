# 工作流整体说明

## 三步工作流

```
输入:一个英文词 (例如 "brainrot")

         ↓

┌────────────────────────────────────┐
│ Step 1: 写作                       │
│ 用 v3-write prompt 调用 OpenAI API │
│ 输入:词                            │
│ 输出:六维骨架解读初稿              │
└────────────────────────────────────┘

         ↓

┌────────────────────────────────────┐
│ Step 2: 查证 + 覆盖审计(Agent)     │
│ 用 v3-research prompt 调用 OpenAI  │
│ 模型在此步骤需要调用工具:           │
│   - web_search (查事实与义项地图)  │
│ 输入:Step 1 的初稿                 │
│ 输出:查证报告 + sense coverage 表  │
└────────────────────────────────────┘

         ↓

┌────────────────────────────────────┐
│ Step 3: 改写                       │
│ 用 v3-rewrite prompt 调用 OpenAI   │
│ 输入:Step 1 初稿 + Step 2 报告     │
│ 输出:改写后的终稿 + 改动说明        │
└────────────────────────────────────┘

         ↓

输出:三份 markdown 文件
```

## 三步之间的数据流

**Step 1 → Step 2:**
- Step 2 的输入是 **Step 1 的完整输出 + 词本身**
- Step 2 不需要原始 Step 1 prompt(避免污染)

**Step 2 → Step 3:**
- Step 3 的输入是 **Step 1 的初稿 + Step 2 的查证报告**(两者拼接给模型)
- Step 3 需要看到 Step 1 完整内容,知道改什么;需要看到 Step 2 完整报告,知道怎么改

## 内容质量闸门

这一步是从 Issue 003 的 `pull / head / render` 修复里加进来的。以后每期都必须跑,不能只靠读稿感觉。

**Gate A:结构完整性**

目的:防止页面出现空段、占位句、标题写错导致渲染器拿不到内容。

每个线上词条至少必须有这些固定标题:

- `**字面含义** · `
- `**本体质感** · `
- `**寻根溯源** · `
- `**各路用法** · `
- `**悟道时刻** · `

如果某个词天然不适合完整展开,也不能省掉标题;要在该段里明确说明为什么不展开。构建前用脚本扫描 Issue 当期词条,任何缺段都不能上线。当前可执行检查:

```bash
python3 word-sense-workflow/audit_content.py --issue "Issue 003"
```

这个脚本只负责机器能判断的部分:固定标题、占位文案、来源列表。它不能替代 sense coverage 审计。

**Gate B:sense coverage 覆盖审计**

目的:防止文章只把初稿写得漂亮,却漏掉真实世界里高频、可查、用户会遇到的用法。

Step 2 不只是查证初稿里的断言,还必须额外做一张“义项地图”:

- 至少对照 2-3 个主流词典或语料来源(Merriam-Webster、Cambridge、Oxford Learner's、OED、Etymonline 等)
- 列出这个词的高频现代义项、固定搭配、短语动词、语域分化和行业用法
- 标注初稿已经覆盖了哪些,漏掉了哪些
- 对漏掉项做决定:必须补 / 可选补 / 有意不补
- “有意不补”必须写理由,例如低频、偏专业、和本期语境无关、会冲淡主线

典型漏项要特别警觉:

- 基础动词的短语动词或现代转义,如 `pull the ad / pull the invite`
- 身体词的计量、职位、方向、心理空间等延展,如 `head count / head to`
- 技术词和传统词义的并存,如 `render page / render a poem`
- 法律、平台、公司条款里的权限结构,如 `at its discretion`
- 新词或网络词的真实使用场域,不要只讲构词趣味

Step 3 必须把 Step 2 标为“必须补”的漏项整合进终稿。不能因为初稿没写,就不补。

## 关键约束

**Step 2 必须有工具调用能力。**

这是工作流的核心。没有工具调用,Step 2 只能让 AI 标注"我不确定",但不能真的去查证。

当前实现使用 OpenAI SDK 的 Responses API,Step 2 通过内置 `web_search` 工具查证。实现里可配置:

- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `OPENAI_MODEL`
- `OPENAI_RESEARCH_MODEL`
- `OPENAI_REWRITE_MODEL`
- `OPENAI_SEARCH_CONTEXT_SIZE`

**Step 3 不需要工具调用,纯文本输入输出。**

## 失败处理

每一步都可能失败:

- **Step 1 失败**:报错,整个工作流停止。让用户重试。
- **Step 2 失败**:有两种情况:
  - 完全失败(API 错误):停止,让用户重试
  - 部分失败(某些事实查不到):**仍然继续 Step 3**,但在 Step 2 报告里明确标注 "无法验证" 的项
- **Step 3 失败**:报错,但保留 Step 1 和 Step 2 的输出,用户可以手动改写

## 时间和成本预期

每个词的完整工作流大约:

- Step 1: 10-30 秒(纯文本生成)
- Step 2: 1-3 分钟(取决于查证调用次数,平均 3-5 次工具调用)
- Step 3: 10-30 秒

**总计每个词约 2-4 分钟,API 成本取决于实际使用的 OpenAI-compatible 模型和搜索上下文大小。**

## 模型选择建议

当前默认:

- **Step 1**:`openai/gpt-5.4-mini`。负责出六维骨架和第一版表达,成本要可控。
- **Step 2**:`openai/gpt-5.4`。负责查证词源、时间线、用法来源,并启用 web_search。
- **Step 3**:`openai/gpt-5.4`。负责保留文章声音,同时按查证结果改写。

`openai/gpt-5.5` 可以作为少量重点词条的主编级复核模型,不作为默认批量生产模型。
